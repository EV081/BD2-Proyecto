PROYECTO 1:
SIMULADOR DE GESTOR DE BASE DE DATOS CON ORGANIZACIÓN
EFICIENTE DE ARCHIVOS
1. Introducción
El presente proyecto tiene como objetivo que el alumno comprenda y aplique de forma integral las técnicas de
organización eficiente de archivos en memoria secundaria, elemento central del componente de acceso a disco
en sistemas gestores de bases de datos (SGBD). A diferencia del uso de librerías de alto nivel, el proyecto exige
implementar desde cero las estructuras de indexación, de modo que el alumno interiorice el costo real de cada
operación en términos de accesos a página y tiempo de ejecución.
El sistema resultante constituye un mini-gestor de bases de datos que integra: (i) estructuras de indexación
clásicas sobre memoria secundaria, (ii) soporte para datos espaciales mediante un índice R-Tree, (iii) un parser
SQL de subconjunto restringido construido con técnicas de compiladores, (iv) un simulador de control de
concurrencia, y (v) una interfaz gráfica de usuario. La evaluación experimental de las técnicas forma parte
obligatoria del proyecto.

2.2 Parser SQL Personalizado
Se debe implementar un parser para un subconjunto de SQL utilizando técnicas de compiladores (autómatas
finitos y gramáticas formales) vistas en cursos previos. El parser debe cubrir, como mínimo, las siguientes
sentencias:
CREATE TABLE <nombre> (<col> <tipo> [INDEX <técnica>], ...) [FROM FILE <path>];
SELECT * FROM <tabla> WHERE <col> = <valor>;
SELECT * FROM <tabla> WHERE <col> BETWEEN <v1> AND <v2>;
SELECT * FROM <tabla> WHERE <col> IN (POINT(<x>, <y>), RADIUS <r>);

2.3 Simulador de Acceso Concurrente
Para cubrir el componente de transacciones, se debe implementar un simulador básico de concurrencia que
contemple:
– Ejecución concurrente de al menos dos transacciones simultáneas sobre las mismas estructuras de índice.
– Registro de operaciones (log simplificado) para ilustrar el orden de ejecución e identificar posibles conflictos.
Opcional (+2 puntos adicionales): Implementar mecanismos de control de concurrencia (bloqueos a nivel de
página shared/exclusive o protocolo de marca de tiempo) y detección de interbloqueos (deadlock detection o
timeout). Se debe documentar la estrategia elegida y demostrar su funcionamiento con casos de prueba.
2.4 Interfaz Gráfica de Usuario (Frontend)
El frontend puede implementarse en cualquier tecnología: Python con librería gráfica (Tkinter, PyQt, etc.), o
una aplicación web (React, Angular, u otro framework). Debe ofrecer:
– Editor de consultas SQL con resaltado sintáctico básico.
– Visualización tabular de resultados.
– Panel de estadísticas en tiempo real: número de accesos a disco (lectura/escritura) y tiempo de ejecución en
milisegundos para la última consulta ejecutada.
– Visualización gráfica de puntos en el plano para consultas sobre el R-Tree (obligatorio).

3. Consideraciones Técnicas
Lenguaje y tecnología: El backend se implementa exclusivamente en Python. Se debe usar programación
orientada a objetos y, donde corresponda, programación genérica para que los índices soporten cualquier tipo
de clave. No se permite usar motores de bases de datos externos (SQLite, PostgreSQL, etc.) ni bibliotecas de
índices de alto nivel, salvo para el R-Tree según se indica.
Acceso a memoria secundaria: Todas las operaciones de lectura y escritura deben realizarse a nivel de página
de tamaño fijo (ej. 4 KB). El código debe contabilizar cada acceso a disco (read/write) por operación.
Datos de prueba: Se deben utilizar datasets reales obtenidos de Kaggle u otra fuente pública. Los datos deben
cargarse desde archivos CSV mediante el comando CREATE TABLE ... FROM FILE.
Grupos: Máximo 5 integrantes por grupo. El código fuente se aloja en un repositorio GitHub público. Se
evaluará la participación de todos los integrantes a través del historial de commits. El despliegue debe facilitarse
mediante Docker Compose.

5. Informe, Presentación y Entregables
5.1 Informe técnico
Documento escrito en Markdown o LaTeX. Debe incluir: introducción y objetivo, descripción de cada técnica
con sus algoritmos y diagramas, análisis teórico comparativo de accesos a disco, descripción del parser SQL
(gramática y autómatas), resultados experimentales con gráficos y discusión, y capturas de la interfaz gráfica.
5.2 Presentación en clase
Cada grupo expondrá su proyecto en clase. La presentación tiene una duración máxima de 10 minutos,
seguida de una ronda de preguntas. Los grupos deben diseñar sus propias diapositivas (PPT u otro formato).
Todos los integrantes del grupo deben participar en la exposición. La presentación forma parte de la rúbrica de
evaluación.
